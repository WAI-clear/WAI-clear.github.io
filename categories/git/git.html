git几乎所有操作都是在本地完成，本地完成后再进行上传。其中本地三中状态可以进行切换，实现版本控制。

git配置：

  • 初始化配置文件

    每台计算机上只需要配置一次，程序升级时会保留配置信息。可以在任何时候再次通过命令来修改它们

    • /etc/gitconfig文件：包含系统上每一个用户及它们仓库的通用配置。如果使用带有–system选项的git config时，它会从此文件读写配置变量
    • ~/.gitconfig 或 ~/.config/git/config文件：只针对当前用户。可以传递 –global选项让git读写此文件
    • 当前使用仓库的git目录中的config文件（就是.git/config):针对该仓库。

    每一个级别覆盖上一级别的配置，所以.git/config的配置变量会覆盖/etc/gitconfig中的配置变量。

git状态变化周期：

untracked: 未被加入到git管理中，如果删除，通过git将无法修复

git删除文件：

rm:删除untracked文件将无法通过git恢复

git rm: 下次提交时，文件就不再纳入版本管理了。如果删除之前修改过并且已经放到暂存区域的，必须要用强制删除选项-f。这样数据就不能被git恢复，主要用来删除暂存区的内容

git状态简化查看：

git下的git status很详细的展示状态变化，但是如果状态很多，那么直接显示就比较复杂。

可以使用git status -s简化。新添加的未跟踪文件前面又？？标记，新添加到暂存区中的文件前面又A标记，修改过的文件前面有M标记。M出现在右边表示该文件被修改了但是还没有放入暂存区，出现在左边表示该文件被修改了并放入了暂存区。

git分支：

git分支概念：

使用分支意味着可以把工作从开发主线中分离开以免影响开发主线。

git分支实际使用：

假设目前项目需要开发新功能，但是自己目前开发了40%，如果立刻提交会造成别人无法正常工作。如果代码在写完之后一并提交，又丢失每天进度。

解决方法：创建属于自己的分支，别人在原来分支工作，互不影响。直到开发完成合并分支完成项目

git中HEAD介绍：

默认情况下，git创建一条主分支（master)。而HEAD是指向master分支的一个指针。实际上HEAD是指向当前工作分支。

如果创建新分支，HEAD指向新分支。如果修改master主分支，master会指向修改后的主分支，HEAD指向master.

默认分支介绍：

在git中，以master为开发主线，后期分支合并也是直接合并到master主分支。

分支创建：

分支创建原理：

创建新的分支，将HEAD指向新分支，完成之后再合并到主分支

git分支合并：

原理：合并的实质就是将分支内容迁移到master主分支，但是合并完成后，当前工作分支没有改变

git快速合并（Fast-forward):用于主分支内容不变其他分支内容改变

如果在两个分支中同时修改了一个文件，合并就会出现问题，解决方法：不要再主分支中修改。

git策略合并（非快速合并）：

主分支和其他分支修改了不同文件

区别：

默认使用快速合并，也可以痛过 –no-ff指定为策略合并

策略合并（主分支和新分支都修改了）使用三次提交（主分支，新分支，合并分支），快速合并（只有新分支修改了）两次提交（新分支，合并分支）。

github仓库创建：

github是一个面向开源及私有软件项目的托管平台，因为只支持git作为唯一的版本库格式进行托管，故名gitHub

版本控制中，需要有中央服务器进行代码管理。中央服务器可以自行搭建，也可以使用github进行搭建。

在github上编辑文章默认提交到master分支

    配置位置:用户目录下的.gitconfig文件中,可以直接修改文件内容或使用git config命令配置.

git工作流程:

项目经理:

  • 搭建项目框架,把框架发送到服务器
  • 将员工的SSH公钥添加到服务器

普通员工:

  • 生成SSH公钥,发送给项目经理
  • 收到项目经理项目地址,git clone代码
  • 本地创建分支 ,开发代码,把代码发送到远程服务器.
  • 一般情况下,master分支用来保存发布的项目,dev分支用来保存开发过程中用的代码.
